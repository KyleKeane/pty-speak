# pty-speak Emission Protocol — Prior-Art Survey and Design Recommendation

**TL;DR**

- The recommended emission protocol for pty-speak’s linear-text substrate is a **layered seam-hierarchy commit model**: emit at the strongest available seam (OSC 133;C/D prompt boundary → newline → idle quantum → max-bytes → max-time), with a **live-region tail mask** that suppresses overwrite-class bytes from extending the substrate.
- Default cadence parameters: **idle quantum 150 ms**, **max-bytes-per-emit 4 096 bytes**, **max-time-without-emit 2 000 ms**, **live-region debounce 250 ms**, drawn from SSE (3 000 ms reconnect,  ~15 s keep-alive), tail (1 s stat), Logstash (1 s stat / 15-multiple discover),  HuggingFace TextStreamer (last-space heuristic), and Reactor’s 256-element default prefetch, then tuned downward for screen-reader latency.
- Live-region detection should be **reactive on a ranked Williams VT500 event set**: `ESC[?1049h/l` (regime switch), `ESC[2J / ESC[H` (full repaint), bare `\r` without `\n`, `ESC[K` (EL), `ESC[A` / `ESC[<n>D` followed by printable bytes (cursor-up/back rewrite). When any of these touch the current row, the row’s bytes are *replacement candidates*, not appendable substrate.

## 2. Per-system prose

### 2.1 WHATWG HTML5 tokenizer (and html5ever, html5gum, Servo)

The HTML Living Standard’s §13.2.5 specifies a state machine where each state either *consumes* an input character, *re-consumes* it in another state, or *emits* a token. The tree-construction stage runs immediately on every emitted token and may inject characters back into the tokenizer.  Crucially, the spec acknowledges that a token can be **created but implicitly abandoned**: if EOF arrives while the parser is mid-StartTag, the partially built token is discarded silently.  html5ever (Rust) and html5gum implement this exactly; html5gum’s `Emitter` trait is explicitly the abandonment-safe boundary.  Servo’s html5ever fork additionally supports parser-pause flags, mirroring the standard’s “before each step, check parser pause flag”  requirement. The takeaway for pty-speak: **the substrate must distinguish “accumulating” from “emittable”, and must tolerate accumulator state being thrown away** when an alt-screen regime change arrives.

### 2.2 Server-Sent Events (W3C/WHATWG `text/event-stream`)

SSE is the closest direct analogue to pty-speak’s problem: a long-lived text stream split into discrete announcements. The spec (HTML Living Standard §9.2) mandates *line-by-line* parsing where lines end at LF, CRLF, or lone CR;  events are dispatched only when a **blank line** is reached (the data-buffer commit). The MIME type is `text/event-stream`;   comments start with `:`  and are recommended every ~15 s as proxy keep-alives.  The default `EventSource` reconnect interval is **3 000 ms**,  settable per-stream via the `retry:` field.   EventSource implementations (browsers, `eventsource` npm, Yaffle’s polyfill at 45 s default activity timeout)  diverge on idle behaviour, but all share the rule: **never dispatch a partial event**.  For pty-speak this validates the “emit at semantic seam, hold partial buffer indefinitely if necessary, but instrument keep-alive announcements” pattern.

### 2.3 HTTP/1.1 chunked transfer encoding (RFC 9112 §7.1)

Chunked encoding solves an isomorphic problem: stream content of unknown total size as a sequence of length-prefixed buffers. Each chunk is `chunk-size [chunk-ext] CRLF chunk-data CRLF`, ending with a zero-size chunk and optional trailer fields.  The recipient MUST be able to parse it;  mid-chunk delay simply blocks the reader. RFC 9112 §8 specifies that a client receiving an incomplete response — connection closed mid-chunk or chunked decoding failed — **MUST record the message as incomplete**.  This is the explicit-incompleteness contract pty-speak should adopt: emissions carry an “is-final” flag, and the consumer (NVDA, FileLogger, UIA) handles “incomplete” without crashing.

### 2.4 IRC (RFC 1459, RFC 2812, IRCv3 message-tags)

IRC is the most conservative line-buffering precedent. Messages are CRLF-terminated, capped at 512 bytes including the trailer (510 useful),  with IRCv3 tags adding 8 191 bytes for clients.  The modern IRC protocol document (`modern.ircdocs.horse`) gives the implementer’s rule verbatim: “Read the incoming data into a buffer. Only parse and process a message once you encounter the `\r\n` at the end of it.”  Empty messages are silently ignored.  Clients hold partial-line buffers indefinitely — there is no idle timeout for partial data; the only failure mode is the 512-byte ceiling causing strict clients to drop overlong messages from non-compliant peers (Twitch).  For pty-speak this is the pure “wait for the seam, however long it takes” reference point — and a warning that without a max-bytes ceiling, a malicious or buggy producer can freeze the announcement pipeline.

### 2.5 NLP streaming (spaCy, HuggingFace, NLTK)

HuggingFace’s `TextStreamer` (`transformers/src/transformers/generation/streamers.py`) embodies the most relevant heuristic: as tokens arrive from `.generate()`, the streamer accumulates decoded text and emits **only up to the last space character**, holding the trailing fragment until the next token resolves it. Subword-piece tokenizers (BPE, SentencePiece) make this essential — emitting a partial token would speak gibberish. `TextIteratorStreamer` exposes the same logic over a queue with a configurable timeout. spaCy’s `Sentencizer` is rule-based  on `.`/`!`/`?` with abbreviation exception lists; the dependency-parser-based default is more accurate but slower   (parser ~22 606 docs/s vs. senter ~301 519 docs/s on en_core_web_sm).  The dominant failure mode is sentence-mid splits on abbreviations (“Dr.”, “St.”) and back-revision: when later context arrives, the prior sentence boundary may be retroactively wrong. pty-speak inherits the “hold the trailing fragment until a whitespace seam” pattern — but for screen-reader latency, the *idle quantum* must override “wait for whitespace” after some bound.

### 2.6 `tail -f` and friends (GNU coreutils, Logstash, Vector)

GNU `tail -f` since coreutils 7.5 uses **inotify by default** on Linux — events arrive as the file is written, with no polling latency. The undocumented `---disable-inotify` reverts to the legacy `--sleep-interval` poll,   default **1.0 s**. The unit of emission is one line (LF-terminated). Lines without a trailing newline are held until the newline arrives; Vector explicitly does not emit the last line if no newline is present  (issue #18341). Logstash’s file input plugin uses `stat_interval` default 1 s  and `discover_interval` default 15 (a multiplier — every 15 stat cycles, run filename-glob discovery),  and stays in “tail” mode treating lines as event boundaries. Vector’s file source allows a configurable `max_line_bytes`;  when the producer rotates the file, in-flight buffers are flushed at EOF on the old descriptor. Failure mode across all three: **partial trailing line lost on rotation**. pty-speak’s analogue is “command end” (rotation) — when the prompt boundary arrives, the trailing partial bytes must be flushed even without a newline.

### 2.7 OSC 133 (FinalTerm semantic prompts; iTerm2, WezTerm, Ghostty, foot, Contour, Windows Terminal)

OSC 133 is the explicit-seam protocol pty-speak should adopt. The format is `ESC ] 133 ; <Cmd> [; <Params>] ST`,  with four sub-commands: `A` (prompt start), `B` (prompt end / command input start), `C` (command output start),  `D[;<exit>]` (command finished). The original spec was published by the FinalTerm project and is mirrored in iTerm2’s documentation,  WezTerm’s `shell-integration.md`, Ghostty’s source (Bash/Zsh/Fish/Elvish/Nushell shell integration),  and Contour’s `vt-extensions/osc-133-shell-integration` page. Fallback behaviour when OSC 133 is absent: terminals revert to row-based prompt-detection regex (iTerm2’s “Shell Integration installer” prompt regex; WezTerm’s “no semantic zones, use selection heuristic”). The dominant failure mode is **partial OSC 133 emission** — for example, a shell that emits `A` but never emits `D` (Ctrl-C between B and C in many bash integrations, per Ghostty’s source comments),  leaving the terminal believing the command is still running. Foot’s row-based implementation also notes that erasing or overwriting the cell where the OSC was emitted does not remove the prompt mark, because the marker is stored per-row not per-cell. 

### 2.8 VS Code terminal shell integration (OSC 633)

VS Code extends OSC 133 with a custom OSC 633 namespace because Microsoft considered FinalTerm’s protocol underspecified for command-line capture. OSC 633 adds: `E ; <commandline> [; <nonce>]` (explicit command line, hex-escaped), `F`/`G` (continuation start/end), `H`/`I` (right-prompt zones), and `P ; <Property>=<Value>` (terminal properties including `Cwd`, `IsWindows`, `HasRichCommandDetection`).  When the shell does not cooperate, VS Code’s `shellIntegrationAddon.ts` falls back to OSC 133 with a degraded experience, then to **regex prompt detection** with explicit Windows-ConPTY heuristics — its docs note that on ConPTY “the pty handles rendering specially”  so “shell integration sequences that identify the commands in the terminal buffer may be misplaced”,   and “VS Code’s heuristics have kicked in to improve the position of the command decorations”  after each command. The lesson: even with explicit semantic markers, ConPTY can shift the markers’ apparent positions, and a heuristic post-correction is required.

### 2.9 Streaming JSON parsers (oboe.js, clarinet, serde_json)

clarinet (Node, SAX-style, ancestor of oboe)  emits per-event callbacks: `onopenobject`, `onkey`, `onvalue`,  `oncloseobject`, etc. The unit of emission is the smallest structural step. oboe.js layers a JSONPath-like selector on top, emitting whole subtree nodes  only when their closing brace arrives. `serde_json::StreamDeserializer` (Rust) operates over top-level values: arrays, objects, or whitespace-separated scalars. When the buffer is truncated mid-value, `next()` returns an EOF error; the caller can splice new bytes onto `&data[stream.byte_offset()..]` and re-create the deserializer. This is the “explicit retry-from-offset” pattern — the parser exposes its consumed-byte boundary so the caller can resume. The failure mode is silent truncation if the caller does not check the EOF flag.

### 2.10 Reactive-stream backpressure (RxJS, Project Reactor, RxJava)

Reactive frameworks codify the consumer-can’t-keep-up problem with named overflow strategies. Project Reactor’s `FluxSink.OverflowStrategy` enum lists **IGNORE** (yield IllegalStateException), **ERROR** (signal overflow), **DROP** (discard new), **LATEST** (keep newest, drop intermediates), and **BUFFER** (unbounded — OOM risk). `BufferOverflowStrategy` adds **DROP_OLDEST**. Reactor’s default operator prefetch is **256** elements. RxJS provides `bufferTime`, `sample`, `throttleTime`, `debounceTime`, `auditTime`, plus the legacy `pausableBuffered` operator and the modern `controlled` reactive-pull model. RxJava’s `onBackpressureBuffer/Drop/Latest`  mirrors Reactor’s vocabulary. The relevance to pty-speak: a Claude thinking-spinner emitting at 10 Hz is precisely the “fast hot publisher, slow consumer” case. The right strategy is **LATEST** (keep the freshest live-region content, drop the intermediate spinner frames) combined with **DEBOUNCE** at the idle quantum. The failure mode of `LATEST` — perceived reorder — does not apply because spinner frames are by definition equivalent.

## 3. Design questions

### 3.A Recommended boundary protocol

The primary emission strategy is a **seam-hierarchy commit** evaluated on every parser tick, with the following ordered seams (strongest first):

1. **Semantic prompt seam** (OSC 133 `D` command-finished, or `C` command-output-start when transitioning out of prompt): drain everything in the buffer immediately; mark emission `final=true`. This handles `dir`, plain commands that finish promptly.
1. **Newline boundary** (LF or CRLF arrived since last emit): emit at the last seen newline, retain trailing partial bytes. This handles per-line streaming during `dir /s C:\Windows\System32`.
1. **Idle quantum** (no bytes received for *idle_quantum_ms*): emit accumulated bytes, retain nothing. This handles Claude’s text-response streaming at 200–400 chars/sec (gaps between sentences and tool-call boundaries).
1. **Max-bytes ceiling** (buffered bytes ≥ *max_bytes_per_emit*): emit a hard chunk regardless of seam. This is the “tight loop with no newlines” safety valve, modeled on IRC’s 512-byte cap and Reactor’s 256-element prefetch.
1. **Max-time ceiling** (now − last_emit ≥ *max_time_without_emit*): emit whatever is buffered. This is the “Claude thinking, no output, no spinner suppressed bytes either” liveness guarantee; modeled on SSE’s 15 s keep-alive cadence.

The substrate accumulator is **append-only** unless live-region detection (§3.B) marks the current row as overwrite-class, in which case the row’s bytes are held in a separate “tail mask” not committed to the substrate. On emission, the tail mask’s *latest* state is appended (LATEST semantics), and the mask is cleared. Spinner frames at 10 Hz therefore never produce 10 announcements per second — they collapse to one announcement per idle quantum, carrying only the most recent glyph.

Acknowledged fallbacks:

- For `dir /s | more`, the `--More--` prompt is a live-region tail that the user dismisses with space; the tail mask absorbs it, NVDA announces the most recent state once per idle quantum, and the user’s space-bar advance triggers a new chunk after the next newline burst.
- For `npm install`, progress bars are tail-mask suppressed; the final summary arrives as plain newlines and emits via seam #2.
- For tab-completion echo, the overwrite is detected via `\r` or cursor-back, the tail mask captures the new prefix, and emission happens at next idle.

### 3.B Live-region discriminator

Detection should be **reactive** on the Williams VT500 parser’s emitted events, not a per-shell heuristic — heuristics break on ConPTY (per VS Code docs) and on alt-screen apps that share row 0 with prompt output. The Coalescer subscribes to the parser’s event stream and applies the following ranked classifier to each row currently being assembled:

1. **`ESC[?1049h` / `ESC[?1049l`** — alternate-screen enter/exit. This is not a row-level overwrite; it is a regime switch (§3.E). The substrate is checkpointed and the pipeline switches pathway.
1. **`ESC[2J` (ED 2)** or **`ESC[H ESC[2J`** sequence — full-screen erase. The current substrate is *frozen* and a new substrate begins; alternatively, treat as a regime switch into TUI pathway if it occurs without an alt-screen toggle.
1. **Bare `\r` (CR) without immediately following `\n`** — classic carriage return for in-line overwrite. The current row enters tail-mask state. This catches progress bars (`[####    ] 47%\r`).
1. **`ESC[K` (Erase in Line)** — erases from cursor to end of line. Marks the current row as tail-mask. Catches `--More--` prompts and tab-completion erase-and-redraw.
1. **`ESC[A` (CUU, cursor up)** followed by printable bytes on a previously committed row — retroactive rewrite. Mark the *target row* as tail-mask; if the target row was already emitted, the rewrite is announced as a separate “[updated]” annotation rather than retroactively patching the substrate.
1. **`ESC[<n>D` (CUB, cursor back)** followed by printable text — partial-line overwrite. Same handling as #3.
1. **Reverse Index (`ESC M`)** when at row 0 — indicates scroll, not overwrite, but combined with tail-mask state it should not extend the substrate.

The discriminator runs in the Williams VT500 parser’s “Action” hook for ground state and CSI dispatch state; it does not require a separate parsing pass. The output of the discriminator is a per-row enum: `Append | TailMask | Frozen`. Only `Append` rows extend the canonical substrate. Detection is **reactive** (driven by parser events), not **proactive** (no per-shell heuristics like “if shell == bash and prompt regex matches…”). The single heuristic exception is recognising OSC 133 `A`/`B` markers as “prompt zone” — a row inside a prompt zone is implicitly tail-mask while the user is editing, and freezes on `B`.

### 3.E Multi-substrate transitions

The cleanest prior-art analogue for Stream → TUI and TUI → Stream switching is the xterm DEC-private-mode 1049 sequence (`ESC[?1049h` enter, `ESC[?1049l` exit), which composes mode 1047 (alt screen) with mode 1048 (cursor save/restore). xterm’s documentation specifies that mode 1049 *saves the cursor, switches to alt screen, and clears it* on entry, and *restores the cursor and original screen* on exit. tmux and screen treat the alt-screen contents as **ephemeral** — they do not flow into scrollback (per the tmux scrollback documentation: “any output that exceeds the visible part of alternate buffer is lost”). WezTerm and Windows Terminal similarly maintain a separate scrollback for the primary screen and discard alt-screen content on exit.

The terminal-emulator pattern is therefore **drain-checkpoint-swap**:

1. On `ESC[?1049h` (or `47h`, `1047h` legacy), the Stream-pathway Coalescer flushes its pending buffer with `final=true`, emits a “stream-suspended” checkpoint to NVDA / FileLogger / UIA, and freezes the linear substrate.
1. The TUI-pathway substrate (the 30×120 cell grid) becomes canonical.
1. On `ESC[?1049l`, the TUI substrate is discarded (per tmux/xterm convention); the linear substrate resumes from its frozen state with a `regime_switch_drain_ms` settle period before any new bytes are committed.

The streaming-systems analogues (Kafka consumer-commit, Flink barrier checkpoints) confirm the principle: a checkpoint is a moment at which the in-flight state is durable and the substrate-identity changes. For pty-speak specifically:

- The in-flight buffer at the transition point is **drained** (Stream → TUI) — bytes already accumulated belong to the linear substrate and should be announced before the regime change. This matches `tail -f`’s drain-on-EOF behaviour.
- TUI → Stream does **not** drain anything inbound from the alt-screen — that content was canonically the screen, not the linear stream. The screen substrate is dropped (matching tmux/xterm), and the linear substrate continues from where it was frozen. This is the “discard scrollback on alt-screen exit” pattern.
- A failed transition (alt-screen entered without a clean exit, e.g., `vim` killed by signal) is recoverable: the next `\r\n` plus prompt emission in the Stream pathway re-establishes a seam, and the frozen prefix is re-announced with an “[interrupted]” marker.

## 4. Closing recommendation (RFC-ready)

The following bullets are intended for direct lift into the pty-speak emission-protocol RFC.

- **Adopt a seam-hierarchy commit model** for the Stream pathway, evaluated on every Williams VT500 parser tick: `OSC 133;C/D` > newline > idle-quantum > max-bytes > max-time. Each stronger seam pre-empts weaker ones. This mirrors SSE’s blank-line dispatch, IRC’s CRLF rule, and HTTP chunked encoding’s length-prefix authority while degrading gracefully when no shell cooperation is available.
- **Default cadence parameters**: `idle_quantum_ms = 150`, `max_bytes_per_emit = 4096`, `max_time_without_emit = 2000`, `live_region_debounce_ms = 250`, `regime_switch_drain_ms = 500`. Expose all five in the Tomlyn config under a `[coalescer]` section. These are reasoned defaults grounded in `tail -f` (1 s poll), Logstash (1 s stat), SSE (3 s reconnect, 15 s keep-alive), Reactor (256 prefetch), and HuggingFace TextStreamer (last-whitespace heuristic), then biased toward screen-reader latency.
- **Detect live regions reactively from VT500 parser events**, ranked: `ESC[?1049h/l` (regime switch), `ESC[2J` (full erase), `\r` without `\n` (CR overwrite), `ESC[K` (EL), `ESC[A` plus printable (CUU rewrite), `ESC[<n>D` plus printable (CUB rewrite). Each emits a per-row classifier `Append | TailMask | Frozen`; only `Append` extends the canonical substrate. No per-shell regex heuristics — that path leads to VS Code’s ConPTY-misalignment failure mode.
- **Use LATEST semantics on the tail mask**: spinners, progress bars, `--More--` prompts, and tab-completion overwrites collapse to one announcement per idle quantum carrying the most recent state, drawn from Reactor’s `BufferOverflowStrategy.LATEST`. 
- **Treat OSC 133 / OSC 633 markers as advisory, not authoritative**: when present they act as the strongest seam; when absent or out of order (as on Windows ConPTY per VS Code docs), the time/byte seams provide forward progress. Match VS Code’s `IsWindows=True` posture by not assuming sequence positions are exact.
- **Emit explicit `final` and `incomplete` flags** on every chunk handed to NVDA / FileLogger / UIA, mirroring RFC 9112 §8’s incomplete-message contract. Consumers must tolerate `incomplete=true` chunks without crashing.
- **Drain-checkpoint-swap on regime transitions**: on `ESC[?1049h` (or 47h/1047h), flush the Stream buffer with `final=true`, checkpoint, freeze the linear substrate, swap to TUI pathway. On `ESC[?1049l`, drop the TUI substrate (tmux/xterm convention), resume the linear substrate after a `regime_switch_drain_ms` settle period.
- **Never flush mid-CSI**: the Williams VT500 parser owns the “complete-sequence” boundary; the Coalescer must not split an emission across an in-progress escape sequence, mirroring the HTML5 tokenizer’s “implicit-abandonment-only-at-EOF” invariant.
- **Cap the substrate accumulator** at `max_bytes_per_emit × 16 = 65 536` bytes per command zone. Producers exceeding this limit force a forced flush (RFC-9112 incomplete-marker semantics) — this prevents the IRC-style “overlong-line stalls the pipeline” failure.
- **Persist the Final event on disposal**: FileLogger always serializes the trailing buffer when the pipeline shuts down, mirroring HuggingFace TextStreamer’s `end()` flush  and avoiding the HTML5-tokenizer-style silent-abandonment failure.

## 5. Caveats

- The default cadence values are starting points, not measured optima. The maintainer should profile NVDA and JAWS announcement latency on representative hardware; the `idle_quantum_ms` value in particular may need to rise to 200–300 ms for slower synthesisers or eSpeak-NG with high speech rates.
- The OSC 133 / OSC 633 ecosystem is not standardised at any IANA registry; its allocation is informal coordination among terminal authors,  and shell integration scripts vary in correctness (Ghostty’s source notes Bash 4.4+ uses `PS0`, older Bash uses the `bash-preexec.sh` shim with a `DEBUG` trap).  Reliance on these markers should always have time/byte fallbacks.
- VS Code’s documented Windows ConPTY caveat (“shell integration sequences … may be misplaced”)  applies directly to pty-speak; do not assume sequence-position fidelity on the Windows backend.
- The “implicit abandonment” pattern from HTML5 is a design choice, not an implementation forced choice — pty-speak could equivalently announce a partial-buffer marker on shutdown. The RFC should pick one explicitly.
- spaCy’s parser-based sentence segmentation gives ~88.6 F1 on OntoNotes (per the linked Sentencizer documentation) versus ~83.6 for the rule-based senter; for pty-speak’s purposes this is moot because *terminal output is not natural-language prose*, but the analogue principle (rule-based seams may regress on edge cases) suggests adding an integration-test corpus of recorded ConPTY sessions for boundary regressions.
- Reactor’s `BUFFER` strategy with no bound is explicitly documented as an OOM risk; pty-speak’s substrate accumulator must always be bounded, even though the natural-language reading of “linear text byte stream that grows append-only” might suggest otherwise.
